Request Flow — загальна схема

Client │ HTTP ▼ uWebSockets.js (server.ts) ← парсинг TLS/HTTP ▼ Router (vendor/utils/routing) ← prefix + url + method, :params ▼ collectRequestMetadata ← cookies/query/headers/IP синхронно ▼ Rate limit (per-IP × per-route) ← 429 + Retry-After + X-RateLimit-* ▼ Content-Type / body-size guards ← 415 / 413 до читання тіла ▼ readAndParseBody ← JSON / multipart / urlencoded / text / octet ▼ Route validator (ArkType) ← 400/422 при помилці ▼ HttpContext { requestId, logger, httpData, responseData, session, auth } ▼ Middleware chain (kernel.ts) ← session_web → csrf_guard → auth_guard → ... ▼ Controller → Service → Repository → Transformer ▼ sendResponse → res.end()

Ключова особливість uWS

Об'єкт req у uWebSockets.js стає невалідним після першого await.

Тому collectRequestMetadata() у server.ts синхронно знімає «снепшот»:

  • cookies, query, headers, IP
  • усе кладеться у звичайні структури (Map/об'єкти)
  • далі працюємо лише з ними

Саме тому весь подальший pipeline оперує HttpContext, а не raw req/res.

Шари та їх відповідальність

ШарПризначенняПравила
ControllerТранспортЧитає payload/params/files, мапить помилки в HTTP-статуси
ServiceБізнес-логікаОркеструє репозиторії, повертає AppResult<T>
RepositoryМежа БДЛише Drizzle-запити, ніякого HTTP/сесії
TransformerКонтракт відповідіDB row → API payload (bigint→string, ISO дати, приховування секретів)

Controller не знає про SQL. Service не знає про HTTP. Repository не знає ні про те, ні про інше.

Body Size Limits

EnvDefaultContent-Type
APP_MAX_JSON_BODY_SIZE2 MBapplication/json
APP_MAX_MULTIPART_BODY_SIZE50 MBmultipart/form-data
APP_MAX_OCTET_BODY_SIZE50 MBapplication/octet-stream
APP_MAX_BODY_SIZE2 MBurlencoded / text / other
  • 413 — Payload Too Large (перевищення ліміту)
  • 415 — Unsupported Media Type (Content-Type поза allowlist)
  • Перевірки відбуваються до читання тіла — буфер не виділяється даремно

Rate Limiting

  • Sliding window, per-process / in-memory
  • Оголошується на групі або маршруті: { windowMs, maxRequests }
  • Перевірка до читання body
HeaderОпис
X-RateLimit-Limitмаксимум за вікно
X-RateLimit-Remainingзалишок
X-RateLimit-Resetабсолютний Unix timestamp (секунди)
Retry-Afterлише на 429, секунди

⚠️ За N інстансами ефективний ліміт = maxRequests × N. Це плата за відсутність мережевого round-trip.

Middleware — сигнатура

const adminGuard: Middleware = async (context, next) => {
  const userId = context.auth.getUserId()
  if (userId === null) {
    context.responseData.status = 403
    context.responseData.payload = { status: 'forbidden', message: 'Admin access required' }
    return   // ← НЕ викликаємо next() → коротке замикання
  }

  const user = await userRepository.findById(BigInt(userId))
  if (user?.role !== 'admin') {
    context.responseData.status = 403
    context.responseData.payload = { status: 'forbidden', message: '...' }
    return
  }

  await next()  // ← пропускаємо далі
}

Controller — реальний registerByEmail

// src/app/controllers/http/auth/auth-controller.ts
export default {
  async registerByEmail(
    context: HttpContext<RegisterEmailInput>,
  ): Promise<RegisterEmailResponse> {
    context.logger.info('registerByEmail handler')
    const payload = getTypedPayload(context)
    const result = await authService.registerByEmail(
      payload,

      context.session,
      context.logger,

    }

    return result.value
  },

}

Транспортний шар у чистому вигляді: лог → getTypedPayload → виклик authService з потрібним контекстом (auth, session, logger, ipAddress) → на помилку mapControllerError, на успіх — пробросити result.value.

mapControllerError — єдиний контракт помилок

// src/app/controllers/shared/controller-error.ts
export function mapControllerError(
  context: Pick<HttpContext<unknown, unknown>, 'responseData'>,
  error: AppError,
): ReturnType<typeof appErrorToHttpPayload> {
  applyAppErrorStatus(context.responseData, error)
  return appErrorToHttpPayload(error)
}
// src/app/services/shared/apply-service-status.ts
export type AppErrorHttpPayload = {
  status: 'error'
  message: string
  code: ReturnType<typeof appErrorCode>
  reason?: string
}

export function applyAppErrorStatus(responseData, error) {
  responseData.status = appErrorStatus(error)
  const retryAfterSeconds = appErrorRetryAfterSeconds(error)
  if (retryAfterSeconds !== undefined) {
    responseData.setHeader('Retry-After', String(retryAfterSeconds))
  }
}

Жоден controller не пише status = 400/401/409 руками. AppError → HTTP status + Retry-After + канонічний payload { status, code, message, reason? } — це той самий контракт, що описано в CLAUDE.md.

Service — AppResult<T>

export type AppError =
  | { _tag: 'BadRequest';   message: string; reason?: string }
  | { _tag: 'Unauthorized'; message: string }
  | { _tag: 'Forbidden';    message: string; reason?: string }
  | { _tag: 'NotFound';     resource: string; message: string }
  | { _tag: 'Conflict';     message: string; reason?: string }
  | { _tag: 'Internal';     publicMessage: string; cause?: unknown }

export type AppResult<T> = Result<T, AppError>
  • Services повертають результат, а не кидають exception
  • tryInternal() / tryDomain() обгортають throw-zones (репозиторії, vendor)
  • Internal.cause — лише для логів/Sentry, у payload не потрапляє

Service — приклад (auth.register)

async register(payload, auth, session) {
  const email = payload.email.toLowerCase()

  const existResult = await tryInternal(
    () => userRepository.findByEmail(email),
    'Failed to load user by email',
  )
  if (Result.isError(existResult)) return Result.err(existResult.error)
  if (existResult.value !== undefined) {
    return Result.err(conflict('Email already exists'))
  }

  const hash = await hashPassword(payload.password)
  const user = await userRepository.create({ name: payload.name, email, password: hash })

  await auth.login(String(user.id), session)
  const wsToken = await generateWsToken(user.id)

  return Result.ok({
    status: 'success',
    user:   userTransformer.serialize(user),
    wsUrl:  getWsUrl(),
    wsToken,
  })
}

Transformer — контракт відповіді

export const userTransformer = {
  serialize(user: UserRow): SerializedUser {
    const { id, password: _p, isAdmin: _a,
            createdAt, updatedAt, emailVerifiedAt, ...rest } = user
    return {
      ...rest,
      id:            String(id),                // bigint → string для JSON
      created_at:    DateTime.fromJSDate(createdAt).toISO(),
      updated_at:    DateTime.fromJSDate(updatedAt).toISO(),
      emailVerified: emailVerifiedAt !== null,
      emailVerifiedAt: emailVerifiedAt === null ? null
        : DateTime.fromJSDate(emailVerifiedAt).toISO(),
    }
  },
}

Саме тут — і лише тут — зрізаються password/isAdmin, конвертується bigint і форматуються дати.

Session & Auth API

// context.auth — interface Auth
getUserId(): string | null
check():     boolean

logout():    Promise<boolean>                 // ← ротує сесію

  • Сховище — Redis, ключ session:{userToken}:{id}, TTL 24h (sliding: GETEX EX поновлює на кожному читанні)
  • SessionData: userId · userToken · wsToken · csrfToken
  • Cookie __Host-uapi у prod (legacy uapi) — httpOnly, secure, SameSite=Lax
  • login/logout/logoutAll ротують session id → захист від session-fixation + свіжий CSRF

CSRF — синхронізуючий токен

  • У кожної Redis-сесії — власний csrfToken
  • login/logout/logout-all ротують сесію та повертають свіжий токен у body
  • main/init також повертає поточний токен
  • Unsafe-методи (POST/PUT/PATCH/DELETE) мають слати X-CSRF-Token
  • csrf_guard реєструється після session_web та до auth_guard
  • Запити без сесії — пропускаються (401 віддасть auth_guard)
  • OPTIONS/safe-методи — пропускаються
  • OAuth redirect/callback — окремо, захист через OAuth state

CSRF у HTML та rollout

<meta name="csrf-token" content="__CSRF_TOKEN__" />

Плейсхолдер замінюється на токен сесії. HTML з токеном — session-bound: Cache-Control: private, no-store. Якщо плейсхолдера немає — лог error, fallback не вставляється.

Env-прапорці

  • CSRF_ENFORCE (default false) — report-only режим для безпечного rollout
  • CSRF_ALLOWED_ORIGINS — список Origin для defense-in-depth
  • CSRF_STRICT_ORIGIN — вимагати обидва заголовки Origin+Referer

Rollout: deploy у report-only → проаналізувати логи → полагодити клієнтів → вмикати enforce поступово.

Trusted proxy / client IP

  • TRUST_PROXY — вмикає довіру до X-Forwarded-For / X-Real-IP
  • TRUSTED_PROXY_CIDRS — IPs/CIDRs, кому дозволено надсилати ці заголовки

Усі auth-throttles працюють за resolved client IP. Неправильна довіра до proxy = послаблений захист від abuse.

proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Real-IP $remote_addr;

Контракт: origin не має бути доступним напряму з інтернету; proxy append-ять, а не replace-ять XFF. X-Real-IP довіряється лише якщо peer ∈ TRUSTED_PROXY_CIDRS.

Phone auth / SMS — fail-fast

  • AUTH_PHONE_ENABLED — вмикає phone flows
  • AUTH_SMS_PROVIDER: unsupported (safe default) | fake (test only)
  • AUTH_OTP_HMAC_KEY — HMAC для хешування OTP у БД (fallback на APP_KEY), ≥32 символи

Fail-fast політика

  • fake провайдер відхиляється без явного AUTH_SMS_PROVIDER_ALLOW_FAKE=true
  • fake відхиляється у production-like (APP_IS_PRODUCTION)
  • Потрібен явний AUTH_FAKE_SMS_CODE
  • Phone auth відхиляється, якщо ключ <32 символів

+ throttles per phone / per IP / per hour для всіх channels.

OAuth (Arctic) — Google / GitHub / GitLab / Facebook

// redirect handler — GET /oauth/:provider
const { url, state, codeVerifier } =
  oauthService.getAuthorizationUrl(provider)        // Arctic + PKCE

await context.session.updateSessionData({           // ← state у сесію
  oauthState: state,
  oauthCodeVerifier: codeVerifier,
})
context.responseData.status = 302
context.responseData.setHeader('Location', url)
// callback handler — GET /oauth/:provider/callback
const storedState   = session.sessionInfo?.data.oauthState
const codeVerifier  = session.sessionInfo?.data.oauthCodeVerifier
// порівнюємо storedState ⟷ receivedState, перевіряємо code
const result = await oauthService.handleCallback(
  provider, code, storedState, receivedState, codeVerifier, auth)
await session.updateSessionData({ oauthState: undefined,
                                  oauthCodeVerifier: undefined })  // cleanup
responseData.setHeader('Location', result.value.redirectUrl)      // 302

Захист — state + PKCE codeVerifier у сесії (тому CSRF-guard пропускає OAuth). Будь-яка помилка → 302 на OAUTH_FRONTEND_ERROR_URL, успіх → OAUTH_FRONTEND_SUCCESS_URL. Identity лягає в oauth_accounts, далі auth.login().

  • VAPID ключі + subject; ensureVapidConfigured() — fail-soft: немає ключів → warn + push вимкнено, без падіння
  • skipIfOnline питає wsConnectionRegistry.isUserOnline() — WS і push не дублюють одне одного
  • Підписки — таблиця push_subscriptions; кожна відправка логується в push_notification_logs (sent / failed)
  • Публічний VAPID-ключ віддається фронту через getVapidPublicKey()

APP_IS_PRODUCTION vs APP_ENV

APP_ENV

  • Описова назва
  • local / dev / staging / prod / prod-eu / live ...
  • Для логів та загального конфігу

APP_IS_PRODUCTION

  • Явний security-класифікатор
  • Для fail-fast перевірок
  • Вмикається навіть за кастомних назв

⚠️ Ніколи не покладатися на парсинг APP_ENV для security-політик. У Docker/Helm/CI виставляти APP_IS_PRODUCTION=true для кожного production-like середовища явно.
